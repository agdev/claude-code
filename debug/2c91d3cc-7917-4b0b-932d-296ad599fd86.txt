##  CATASTROPHIC RISKS (Must Fix Before Implementation)

### Risk #1: Double Initialization Hell

**Probability**: 70%  
**Impact**: Data corruption, duplicate API calls  
**Severity**: CRITICAL 

#### The Problem

Both `OfflineProvider` and `useOfflineWords` hook initialize their own instances of `SyncManager`. This creates:

1. **Two SyncManager instances running simultaneously**
2. **Duplicate event listeners** for network changes
3. **Duplicate sync operations** (both will sync on reconnect)
4. **Race conditions** when both try to modify storage

#### Evidence

**File**: `/game/src/providers/OfflineProvider.tsx` (line 47)
```typescript
const syncManager = SyncManager.getInstance();
// Initializes SyncManager
```

**File**: `/game/src/hooks/useOfflineWords.ts` (line 23)
```typescript
const syncManager = SyncManager.getInstance();
// ALSO initializes SyncManager
```

Both get the same singleton instance BUT both call `start()` and add event listeners.

#### What Will Happen

1. User opens app  OfflineProvider starts SyncManager
2. WordsScreen renders  useOfflineWords ALSO starts SyncManager
3. Network reconnects  **BOTH trigger sync simultaneously**
4. Both try to write to AsyncStorage  **data corruption**
5. API gets hit twice for same data  **wasted bandwidth**

#### Mitigation (REQUIRED)

**Option A**: Remove SyncManager from useOfflineWords entirely
- Only OfflineProvider should manage sync
- useOfflineWords should only read/write local storage
- Use OfflineProvider's sync trigger for manual sync

**Option B**: Add initialization guard in SyncManager
```typescript
private isStarted = false;

async start() {
  if (this.isStarted) {
    console.warn('SyncManager already started');
    return;
  }
  this.isStarted = true;
  // ... rest of start logic
}
```

### Risk #3: Memory Leak Pattern

**Probability**: 100%  
**Impact**: Battery drain, eventual app crash  
**Severity**: HIGH 

#### The Problem

Event listeners are added but **never removed**. This creates memory leaks that accumulate over time.

#### Evidence  

**File**: `/game/src/services/SyncManager.ts` (line 89)
```typescript
NetInfo.addEventListener(state => {
  // Event listener added
  // NEVER REMOVED
});
```

**File**: `/game/src/providers/OfflineProvider.tsx` (line 67)
```typescript
useEffect(() => {
  const subscription = syncManager.on('sync_completed', handleSyncComplete);
  // No cleanup function
  // Listener persists forever
}, []);
```

#### What Will Happen

1. User navigates between screens multiple times
2. Each mount adds new event listener
3. Listeners accumulate: 1  5  10  50
4. Battery drains from excessive background activity
5. Memory usage grows unbounded
6. Eventually: App crashes from memory pressure

#### Mitigation (REQUIRED)

Add cleanup everywhere:
```typescript
useEffect(() => {
  const unsubscribe = syncManager.on('event', handler);
  
  return () => {
    unsubscribe(); // CRITICAL: Remove listener on unmount
  };
}, []);
```

### Risk #4: Authentication Provider Mismatch

**Probability**: 90%  
**Impact**: All existing user data becomes invisible  
**Severity**: CRITICAL 

#### The Problem

####Evidence

```typescript
const user = { id: 'test_user_1' };
```

**Words stored with**:

**After switching to Clerk**:

```typescript
const user = { id: 'user_2abc123xyz' }; // Real Clerk ID
```

```typescript
.eq('clerk_user_id', 'user_2abc123xyz')

```

#### What Will Happen

1. Users have words in database with mock IDs
2. App switches to real Clerk auth
3. User IDs change from `test_user_1` to real Clerk IDs
4. ALL user data becomes inaccessible
5. Users see empty app, think data was deleted
6. Support tickets flood in

#### Mitigation (REQUIRED)

**Before launch**:
1. Use real Clerk from day one (remove TestingAuthProvider)
2. OR: Add migration script to update user_ids
3. OR: Clear all test data before production launch

**Migration script**:
```sql
-- Map test users to real users
UPDATE words

WHERE clerk_user_id = 'test_user_1';
```

## DATA INTEGRITY RISKS

### Risk #5: AsyncStorage Failures (Unhandled)

**Probability**: 30%  
**Impact**: Total data loss  
**Severity**: CRITICAL 

#### The Problem

AsyncStorage can fail in multiple ways, all unhandled:

1. **6MB quota on Android** - Will be hit at ~7,500 words
2. **JSON corruption** - App crashes during write
3. **Storage full** - Device runs out of space

**Add quota checking**:
